# 여름 휴게소 충전 이벤트 페이지 — 웹뷰 연동 개발 스펙

> 목적: 이 웹페이지를 **일렉링크 앱 이벤트 페이지 "참여하기" 클릭 시 웹뷰**로 띄우기 위한 스펙.
> 앱은 이미 로그인된 **회원 토큰**을 가지고 있고, 웹뷰는 그 회원이 누구인지 인지해 **이벤트 참여를 기록**해야 함.
> 이 문서는 (1) 현재 웹페이지 구조(As‑Is)와 (2) 웹뷰 연동에 필요한 추가 작업(To‑Be)을 정리한다.

## 1. 현재 구조 (As‑Is)

### 1.1 호스팅 · 페이지 구성
- **정적 사이트**. 서버 로직 없음. GitHub Pages가 **레포 루트**를 서빙(브랜치 `master`).
- 페이지 3종
  | 파일 | 역할 | 비고 |
  |---|---|---|
  | `index.html` | 이벤트 랜딩(프로모션) | 지도앱을 `<iframe src="app.html">`로 임베드 |
  | `app.html` | 경로 탐색 지도앱(핵심) | Kakao 지도, 하단 "경로 찾기" 시트 |
  | `sk_chargers.json`, `food.json` | 지도앱 데이터 | 빌드 산출물(아래 1.3) |
- **로컬 개발본**: `route/index.html`(= app.html 소스, `__JSKEY__` 플레이스홀더), `route/promo.html`(= index.html), `route/server.py`(로컬 서버·Kakao 길찾기 프록시). 배포본 루트 `app.html`은 `route/index.html`에서 키·경로만 치환해 생성.

### 1.2 클라이언트 동작
- Kakao Maps JS SDK 로드(appkey는 HTML에 인라인). **도메인 화이트리스트 검증** 있음 → 등록 안 된 도메인에서 지도 안 뜸(중요, 3.2 참조).
- 출발지·도착지 검색 → 경로 폴리라인 + 경로 주변 SK 충전소 마커 표시.
- **로그인/회원 개념 없음.** 참여 "기록"도 없음 — 랜딩 카피의 "경로를 한 번 찾으면 응모 완료"는 안내 문구일 뿐, 실제 서버 적재는 **미구현**.
- 클라이언트 저장소: `localStorage`(최근 검색 기록 `route_history`)만 사용.

### 1.3 데이터 파이프라인 (빌드 타임, 앱 런타임과 무관)
- `route/build_sk.py`: `SK일렉링크_..._현황.ver3_주말혼잡도.xlsx` → `sk_chargers.json`(91개소=운영72+설치예정19, 응모권·충전단가·주말혼잡도 포함).
- `route/build_food.py`: 도로공사(EX) OpenAPI → `food.json`(대표메뉴). 실행에 `EX_API_KEY` 필요.
- `build_geocode.py`: 설치예정 좌표 지오코딩(Kakao REST, `.env`의 `KAKAO_REST_KEY`).

### 1.4 외부 의존성 (웹뷰에서 반드시 허용되어야 함)
| 대상 | 용도 | 호출 주체 |
|---|---|---|
| `dapi.kakao.com` (Kakao Maps JS SDK) | 지도 타일·지오코딩·장소검색 | 브라우저(JS SDK) |
| `highway-ev-route-proxy.ian-1f3.workers.dev` | `/route`(Kakao 길찾기 프록시), `/blogs`(네이버 블로그) | fetch |
| (프록시 뒤) Kakao Mobility, Naver 검색 API | 길찾기·리뷰 | Cloudflare Worker |

> 키 은닉: REST 키(길찾기·네이버)는 Cloudflare Worker에 환경변수로 보관. JS 키만 클라이언트 노출(불가피, 도메인 제한으로 보호).

## 2. 연동 목표 (To‑Be)

```
[일렉링크 앱 · 이벤트 페이지]
        │  "참여하기" 탭
        ▼
[웹뷰 오픈]  ──(회원 토큰 전달)──▶  [이 웹페이지]
                                        │  ① 토큰 수신·보관(메모리)
                                        │  ② 참여 트리거 시 백엔드로 참여 기록 요청
                                        ▼
                              [일렉링크 백엔드]  ── 토큰검증 → 회원 식별 → 참여 적재
                                        │
                                        ▼
                              성공/실패 응답 → 웹에서 UI 처리(완료 토스트 등)
```

핵심 신규 작업 3가지:
1. **회원 토큰을 앱 → 웹뷰로 전달**(§3.1)
2. **참여 기록 백엔드 API 신설**(§3.3) — *현재 백엔드 없음*
3. **웹뷰 환경 요구사항 충족**(§3.2)

## 3. 상세 스펙

### 3.1 회원 토큰 전달 (앱 → 웹뷰)

두 가지 방식. **권장: A(URL 파라미터) + B(JS 브리지) 병행** — A로 초기 진입을 단순화하고, 토큰 만료 시 B로 재발급.

#### 방식 A — URL 쿼리 파라미터 (권장·1차)
앱이 웹뷰 URL에 **단기 토큰**을 실어 연다.
```
https://<이벤트도메인>/?tk=<ACCESS_TOKEN>&uid=<선택: 회원해시>&ch=app
```
- `tk`: 서버 검증용 액세스 토큰(불투명 문자열, 짧은 TTL 권장 5~15분).
- `ch=app`: 진입 채널 구분(웹 직접 진입과 구별, 통계용).
- 웹 처리(예시):
  ```js
  const q = new URLSearchParams(location.search);
  const token = q.get("tk");            // 메모리에만 보관 (localStorage 저장 금지)
  // 주소창/기록 노출 최소화: 읽은 뒤 히스토리에서 제거
  history.replaceState(null, "", location.pathname);
  ```
- 장점: 앱 구현 간단(웹뷰 URL만 구성). 단점: URL 노출 → **짧은 TTL·1회성** 필수.

#### 방식 B — 네이티브 JS 브리지 (권장·보강)
웹이 필요 시 토큰을 능동 요청. 앱이 webview에 인터페이스 주입.
- Android: `webView.addJavascriptInterface(obj, "ElecLink")` → `window.ElecLink.getToken()`
- iOS(WKWebView): `WKScriptMessageHandler` + `window.webkit.messageHandlers.ElecLink.postMessage(...)` (콜백은 `window`에 함수 심어 회신)
- 웹 공통 어댑터(예시):
  ```js
  async function getMemberToken(){
    const q = new URLSearchParams(location.search).get("tk");
    if (q) return q;                                   // A 우선
    if (window.ElecLink?.getToken) return window.ElecLink.getToken();      // Android
    if (window.webkit?.messageHandlers?.ElecLink){     // iOS
      return new Promise(res=>{ window.__onToken=res; window.webkit.messageHandlers.ElecLink.postMessage("getToken"); });
    }
    return null;                                       // 웹 직접 진입(비회원)
  }
  ```
> 앱팀 결정 필요: 인터페이스 이름(`ElecLink`), 메서드 시그니처, iOS 콜백 규약.

### 3.2 웹뷰 환경 요구사항

- **Kakao JS 키 도메인 등록(필수)**: 웹뷰가 로드하는 **최종 도메인**을 Kakao Developers → 앱 → 플랫폼 → Web 사이트 도메인에 등록. 미등록 시 SDK 401(`domain mismatched!`)로 **지도 안 뜸**.
  - 현재 배포 도메인(GitHub Pages)과 다른 도메인(예: 일렉링크 도메인)으로 서빙한다면 그 도메인도 등록.
  - `file://` 직접 로드 불가(도메인 검증 실패). 반드시 `https://`.
- **WebView 설정**
  - JavaScript 활성화(Android `settings.javaScriptEnabled=true`).
  - DOM Storage 활성화(Android `domStorageEnabled=true`) — `localStorage`(최근검색) 사용.
  - HTTPS 전용(Mixed content 차단 유지). 외부 호출 도메인 §1.4 허용.
  - iOS `WKWebView` 권장(구형 `UIWebView` 금지). `allowsInlineMediaPlayback` 등 불필요.
  - 위치권한 불필요(현재 GPS 미사용).
- **레이아웃**: 하단 안전영역 `env(safe-area-inset-bottom)` 이미 사용. 웹뷰 높이는 풀스크린 권장(지도가 `position:fixed; inset:0`).
- **뒤로가기/닫기**: 웹 내부 라우팅 없음(단일 페이지). 앱의 상단 닫기 버튼으로 종료. 참여 완료 후 앱으로 신호가 필요하면 §3.4.

### 3.3 참여 기록 백엔드 계약 (신규 · 일렉링크 백엔드에 구현 필요)

현재 참여를 적재할 서버가 없다. 아래 **엔드포인트 1개** 신설 제안.

**요청**
```
POST https://<elec-link-api>/event/summer-charge/participate
Authorization: Bearer <ACCESS_TOKEN>        # 또는 body의 tk
Content-Type: application/json

{ "channel": "app-webview", "eventId": "summer-2026", "clientTs": "2026-07-02T09:00:00+09:00" }
```
**응답**
```json
{ "ok": true, "memberId": "m_xxx", "alreadyParticipated": false, "participatedAt": "2026-07-02T09:00:01+09:00" }
```
- 서버: 토큰 검증 → 회원 식별 → **멱등 처리**(동일 회원 재요청 시 `alreadyParticipated:true`, 중복 적재 금지).
- 실패: `401`(토큰 만료/무효 → 웹은 §3.1‑B로 재발급 시도), `409/200+alreadyParticipated`(이미 참여).
- **CORS**: 웹 도메인 오리진 허용 필요(`Access-Control-Allow-Origin`).

> "참여" 트리거 시점(앱팀·기획 확정 필요): (a) 웹뷰 진입 즉시, (b) "경로 찾기" 1회 완료 시, (c) 별도 "참여하기" 버튼. 현재 랜딩 카피는 "경로 1회 탐색 = 응모 완료" 기준 → (b) 권장.

### 3.4 (선택) 참여 완료 → 앱 콜백
참여 성공 후 앱 UI(배지/토스트/닫기)를 위해 신호가 필요하면 §3.1‑B 브리지 역방향 사용:
```js
window.ElecLink?.onParticipated?.(JSON.stringify({ ok:true }));         // Android
window.webkit?.messageHandlers?.ElecLink?.postMessage({ type:"participated" }); // iOS
```

### 3.5 보안 유의
- 토큰은 **메모리 보관**, `localStorage`/URL 잔존 금지(진입 후 `history.replaceState`로 제거).
- 토큰 **짧은 TTL + 1회성**, 참여 검증·적재는 **서버에서만**(클라 신뢰 금지).
- 로그/에러 리포팅에 토큰 미노출. 전 구간 HTTPS.

## 4. 앱팀과 확정할 사항 (체크리스트)

- [ ] 최종 웹뷰 서빙 **도메인** 확정 → Kakao JS 키 도메인 등록 (그대로 GitHub Pages 사용 여부 포함)
- [ ] 토큰 전달 방식: URL 파라미터(A) / JS 브리지(B) / 병행 — 인터페이스 이름·시그니처
- [ ] 토큰 형식(JWT? 불투명?)·TTL·검증 방법
- [ ] 참여 기록 API 스펙 확정(경로·요청/응답·CORS·멱등) 및 구현 주체
- [ ] "참여 완료" 트리거 시점(진입/경로탐색/버튼)
- [ ] 참여 완료 후 앱 콜백 필요 여부(§3.4)
- [ ] 비회원(웹 직접 진입) 처리 정책 — 안내만 vs 가입 유도

## 5. 참고 — 현재 배포/빌드 요약
- 배포: `master` push → GitHub Pages(**레포 루트**) 자동 반영. 데이터/문구 수정은 `route/*` 수정 후 루트로 반영·커밋.

- 키: `.env`(커밋 금지)에 `KAKAO_JS_KEY`(지도), `KAKAO_REST_KEY`(지오코딩·길찾기), `EX_API_KEY`(메뉴). 클라이언트 노출은 JS 키뿐.
